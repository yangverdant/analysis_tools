# 云端同步与数据资产整改方案

## 一、结论

云服务器数据需要整改，但不是用本地数据库覆盖云端数据库。

当前判断是：云端数据库更接近生产资产，本地数据库更适合开发和验证。后续同步应以代码、前端构建产物、迁移脚本为主；数据层只做备份后的结构迁移、增量回填和质量修复。

## 二、为什么不能直接覆盖云端库

本地库已经有新的情报表和训练样本表，适合验证新功能；但云端库积累了更多真实运行数据、赛果、验证记录和线上业务状态。

如果用本地 `data/football_v2.db` 覆盖云端 `/opt/football_tools/data/football_v2.db`，风险是：

1. 云端历史比赛、赛果、验证记录被回退。
2. 已经运行过的生产分析和复盘样本丢失。
3. 体彩中心线上展示和调度状态出现断层。
4. 未来训练集会混入错误版本，导致模型判断不稳定。

所以云端数据整改的正确方式是：保留云端主库，在云端主库上补结构、补索引、补数据质量。

## 三、同步原则

1. 本地负责开发：改代码、改前端、跑测试、做构建。
2. 云端负责生产：保存长期数据、运行调度、持续采集、沉淀训练样本。
3. 同步代码时排除本地数据库、导出文件、日志和临时缓存。
4. 所有数据库结构变化必须通过可重复执行的迁移或初始化函数完成。
5. 所有历史回填都在云端现有库上增量执行。
6. 每次云端操作前先备份数据库。

## 四、推荐云端上线流程

### 1. 本地确认

本地先完成：

- 前端构建通过。
- 后端启动无导入错误。
- `/api/v1/intelligence/health` 本地可访问。
- 体彩中心能正常展示比赛卡片。
- 情报证据状态不会影响无情报表的旧环境。

### 2. 云端备份

在云服务器执行：

```bash
mkdir -p /opt/football_tools/backups
cp /opt/football_tools/data/football_v2.db /opt/football_tools/backups/football_v2_$(date +%Y%m%d_%H%M%S).db
```

### 3. 同步代码和前端构建产物

同步时必须排除：

- `data/*.db`
- `data/*.sqlite`
- `data/intelligence_exports/*`
- `logs/*`
- `backups/*`
- `node_modules/*`
- `.git/*`

建议只同步：

- `backend/`
- `frontend/dist/`
- `fetchers/`
- `scripts/` 中明确仍在使用的运营脚本
- `config/`
- `deploy/`
- `*.md` 架构和运维文档

### 4. 云端结构迁移

云端不换库，只在原库上补齐新结构。当前重点是让云端拥有情报层所需表：

- `intelligence_jobs`
- `intelligence_requirements`
- `intelligence_artifacts`
- `intelligence_packages`
- `intelligence_reviews`
- `intelligence_runs`

这些表应由后端情报模块的 storage 初始化逻辑创建，不能靠复制本地 db 文件。

### 5. 云端重启与验证

重启服务后验证：

```bash
curl http://localhost:8000/api/v1/health
curl http://localhost:8000/api/v1/lottery/health
curl http://localhost:8000/api/v1/intelligence/health
curl http://localhost:8000/api/scheduler/status
```

如果 `/api/v1/intelligence/health` 仍然 404，说明云端代码或路由没有部署成功。

### 6. 云端增量回填

结构确认后，再在云端生产库上执行：

1. 生成今天/近期比赛的 intelligence jobs。
2. 对已结束比赛批量生成 package 和 review。
3. 先不开放大规模联网回填，优先使用云端已有数据和本地 fallback。
4. 再逐步打开天气、新闻、伤停、预计首发等外部采集。

## 五、数据层整改重点

### 1. 情报表要补到云端

云端当前体彩接口可用，但情报接口缺失。第一步不是迁数据，而是让云端代码和表结构具备情报能力。

### 2. 历史比赛要变成训练样本

已结束比赛的数据已经定型，应该逐步回填为：match package、赛果 review、错误归因、训练样本。

### 3. 实时数据要保存快照

赔率、新闻、天气、伤停、预计首发在赛前是实时信息；比赛结束后就是历史证据。采集时必须保存 captured_at、source、confidence、payload、是否 fallback。

### 4. 数据质量要显式展示

前端不应该只展示预测，还要展示证据覆盖率、严格覆盖率、缺失项和兜底证据。低覆盖率的预测不能和高覆盖率预测看起来一样。

### 5. 后续再评估数据库升级

短期继续使用 SQLite 可以，但要控制并发写入和大查询。中长期建议把生产业务库、分析训练库、历史原始数据分层：

- 业务 serving：可继续 SQLite 或迁 PostgreSQL。
- 离线分析和特征：DuckDB 或 Parquet。
- 原始采集归档：JSONL/Parquet/对象存储。

## 六、当前项目应该做什么

当前最重要的不是继续堆新页面，而是把体彩中心变成主驾驶舱，把情报层作为证据系统嵌进去。

每日流程应该是：

1. 体彩中心同步今天比赛。
2. 生成每场比赛的情报任务。
3. 按比赛分发给赔率、阵容、伤停、新闻、天气、国家队语境等采集器。
4. 每场生成 match package。
5. 体彩中心卡片展示预测、赛果、证据覆盖和缺失项。
6. 单场详情展示分析结论和证据面。
7. 比赛结束后写入 review 和训练样本。
8. 复盘为什么对、为什么错，再优化模型和采集优先级。

这条线跑稳之后，再把联赛分析作为第二视角扩展。
