# 项目长期架构目标

## 一、项目定位

这个项目的核心不是单次比赛预测，也不是单一数据源采集，而是建设一个长期沉淀、可查询、可分析、可训练、可接入 Agent 的足球数据与分析资产库。

它需要持续集成多种数据渠道，包括官方 API、第三方 API、爬虫、网页前端数据扒取、手动补录、历史数据文件、赔率源、新闻源、阵容伤停源、天气场地源等。不同来源的数据可以质量不同、时效不同、结构不同，但最终都要被纳入统一的数据生命周期：采集、清洗、归一、存储、证据快照、分析、复盘、训练、再优化。

## 二、核心原则

1. 实时数据最终都会变成历史数据。

   赛前赔率、新闻、天气、伤停、预计首发、盘口变化等内容在比赛前是实时信息；比赛结束后，它们就成为这场比赛的历史证据。项目必须把这些证据保存下来，而不是只拿来临时分析一次。

2. 历史数据是模型和分析能力的燃料。

   已结束比赛的赛果、赔率变化、阵容、新闻、状态、模型预测、实际结果、错误归因，都应该沉淀为训练样本和分析样本。后续模型能不能越来越强，取决于这些样本是否完整、可追溯、可复盘。

3. 体彩中心是当前业务主驾驶舱。

   每天用户最直接需要看到的是：今天有哪些比赛、每场预测情况、赔率和玩法、赛后赛果、点击后的详细分析。情报系统、Agent、训练样本、数据采集，都应该服务这个主界面，而不是抢掉主流程。

4. 情报系统是证据层，不是另一个孤立产品。

   情报任务、采集包、证据缺失、覆盖率、复盘归因，应该嵌入单场比赛卡片和详情，告诉用户这场分析依赖了什么证据、缺什么证据、哪些证据是兜底来的、为什么可能对或错。

5. 云端数据库是生产数据资产，不能被本地库直接覆盖。

   本地负责开发、验证和构建；云端负责长期运行和生产数据沉淀。同步云服务器时，代码可以同步，数据库必须备份后迁移和增量更新。

## 三、目标能力

### 1. 多源采集能力

项目要允许多种采集方式并存：

- 稳定 API：官方接口、第三方足球数据接口、赔率接口、天气接口。
- 半稳定 API：移动端接口、网页内部接口、未公开但可复用的请求。
- 网页爬虫：新闻、伤停、阵容、球队动态、赛前信息。
- 前端扒取：页面渲染后的赔率、赛程、统计、事件数据。
- 文件导入：CSV、JSON、历史数据库、oddsfe 宽表、人工整理数据。
- 手动补录：关键比赛缺失信息、特殊事件、赛后复盘备注。

所有采集器都应该输出统一结构：来源、采集时间、目标比赛、原始 payload、归一化字段、置信度、是否兜底、错误信息。

### 2. 数据生命周期管理

每条数据应该能进入以下层次：

- raw：原始响应或原始页面内容。
- staging：初步解析后的半结构化数据。
- normalized：统一实体、时间、状态、字段后的数据。
- serving：供前端和接口直接查询的业务表或视图。
- evidence snapshot：绑定到某场比赛、某次分析时点的证据包。
- training sample：赛后带结果和归因的训练样本。

### 3. 比赛驱动的分析包

每场比赛应该形成一个可复用的 match package，至少包括：

- 基础信息：比赛、球队、联赛、开球时间、数据源 ID。
- 赔率：初盘、即时、收盘、变化、盘口方向。
- 球队状态：近期战绩、进失球、主客表现、赛程密度。
- 国家队/世界杯语境：赛事阶段、同组/同系列比赛、晋级形势、历史交锋。
- 阵容与球员：预计首发、伤停、停赛、关键球员能力、轮换风险。
- 外部环境：天气、场地、旅行、时区、裁判等。
- 新闻和信息差：球队新闻、舆情、训练动态、临场消息。
- 数据质量：缺失项、兜底项、置信度、来源优先级。

### 4. Agent 接入能力

Agent 不应该直接在混乱数据库里自由乱查，而应该通过明确工具访问项目能力：

- 查询比赛和球队。
- 查询单场 match package。
- 查询历史同类比赛。
- 查询某类证据的历史命中影响。
- 查询模型预测与赛果复盘。
- 触发采集、补齐、复盘、导出训练样本。

Agent 的输出也要结构化保存，成为可复盘的分析记录，而不是一次性聊天内容。

### 5. 训练与复盘闭环

每场比赛结束后，系统应该记录：

- 当时预测了什么。
- 当时用了哪些证据。
- 当时缺哪些证据。
- 实际结果是什么。
- 预测是对还是错。
- 错因归属：证据缺失、盘口变化未捕捉、模型判断偏差、赛事语境误判、临场信息缺失等。
- 这条样本是否适合进入训练集。

最终目标不是简单提高一次准确率，而是知道为什么对、为什么错，并把这个过程变成可训练、可校准、可解释的数据资产。

## 四、当前重点

当前阶段优先级应该是：

1. 稳定体彩中心作为主驾驶舱。
2. 稳定世界杯/国家队比赛的采集、分析、复盘。
3. 把情报证据层嵌入体彩中心卡片和单场详情。
4. 云端只做结构迁移和增量回填，不覆盖生产数据库。
5. 把已结束比赛逐步回填为训练样本。
6. 在国家队/世界杯稳定后，再扩展到联赛分析。

## 五、云端数据原则

云端数据库是生产资产。同步流程应遵守：

1. 本地改代码和前端。
2. 本地构建和测试。
3. 云端备份数据库。
4. 同步代码和前端 dist，排除本地数据库。
5. 在云端现有数据库上执行迁移。
6. 用云端现有数据执行增量回填。
7. 验证 API、体彩中心、调度任务、训练样本。

严禁在没有明确确认和备份的情况下，用本地 `data/football_v2.db` 覆盖云端 `/opt/football_tools/data/football_v2.db`。

具体上线和整改流程见 `云端同步与数据资产整改方案.md`。云端只做结构迁移、增量回填、质量修复和调度验证；本地只负责代码开发和构建。

## 六、最终形态

最终项目应该形成一个强大的足球数据和分析库集：

- 能持续采集。
- 能长期存储。
- 能统一查询。
- 能支撑量化分析。
- 能接入 Agent 推理。
- 能对单场比赛形成证据包。
- 能复盘预测对错。
- 能不断沉淀训练样本。
- 能随着历史样本增加而持续优化。

这才是项目真正的核心价值。
