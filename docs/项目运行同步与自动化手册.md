## 1. 这份手册解决什么问题

这份手册把当前项目的真实运行情况、目标状态、云端和本地的同步方式、自动化任务、备份原则、日常检查命令、故障处理方式统一写清楚。

项目当前已经不只是一个“页面 + 数据库”的工具，而是在往一个持续运行的数据采集、分析、复盘、学习系统演进。核心目标不是某一场比赛猜对，而是让每一场比赛都形成可追溯证据、可验证预测、可复盘错误、可迭代模型的长期数据资产。

必须区分三件事：

1. 代码同步：页面、后端、脚本、部署配置，从本地同步到云端。
2. 数据库同步：`football_v2.db`、`oddsfe_merged.db` 这类大文件，不能跟代码包混在一起随便覆盖。
3. 运行时采集：云端自己持续采集、补齐、复盘、学习，不能依赖人工一下一下触发。

### 2.2 云端 API 进程

云端 API 由 systemd 管理：

```bash
sudo systemctl status football-analyst.service --no-pager -l
```

当前关键配置：

```ini
[Service]
Environment=ODDSFE_DB_PATH=/opt/football_tools/fetchers/odds_feed_api/oddsfe_merged.db
Environment=DISABLE_DAILY_SCHEDULER=1
Environment=PYTHONUNBUFFERED=1
```

重点：

- `DISABLE_DAILY_SCHEDULER=1` 是刻意设置的。
- 旧的 FastAPI 进程内 APScheduler 会造成数据库锁和页面卡顿。
- 现在 API 只负责提供接口，自动化任务由独立 systemd timer 承担。
- 不要随便恢复进程内 scheduler。

### 2.4 轻量自动化中控

脚本：

```bash
/opt/football_tools/scripts/cloud_automation_tick.sh
```

本地文件：

```text
scripts/cloud_automation_tick.sh
```

默认行为：

- 每 15 分钟跑一次。
- 使用同一个中控锁：`/tmp/football_automation_center.lock`。
- 使用 `flock -n` 防止重叠。
- 使用 `timeout` 防止卡死。
- 默认跳过重学习：`--no-learning`。
- 主要处理采集、赛果、O/U、情报、分析、验证、错误归因、轻量纠错。
- 写日志到：

```bash
/opt/football_tools/logs/automation/cloud_automation_tick.log
```

典型命令结构：

```bash
python scripts/run_automation_center.py \
  --mode mixed \
  --league 世界杯 \
  --historical-dates 1 \
  --historical-lookback-days 90 \
  --workers 2 \
  --task-timeout 180 \
  --max-events 4 \
  --max-analysis 8 \
  --max-intelligence 4 \
  --max-validation-dates 1 \
  --no-learning \
  --apply
```

设计原则：

- 高频，但轻。
- 不做大规模学习。
- 不阻塞 API。
- 单次任务失败不能卡住整个系统。
- 只要服务器开着，它就会持续补齐今日、明日、已结束比赛和少量历史缺口。

### 2.5 夜间重学习

脚本：

```bash
/opt/football_tools/scripts/cloud_learning_refresh.sh
```

本地文件：

```text
scripts/cloud_learning_refresh.sh
```

默认行为：

- 每天凌晨执行。
- 默认时间：`02:45`，带 `RandomizedDelaySec=20min`。
- 当前下一次计划：`2026-06-27 03:01:22 CST`。
- 使用 `workers=1`。
- 使用 `nice` 和 `ionice` 降低优先级。
- 使用同一个中控锁，避免和 15 分钟轻量 tick 并发抢库。
- 包含 `--force-learning`。
- 会在学习后触发未开赛比赛重新分析。
- 写日志到：

```bash
/opt/football_tools/logs/automation/cloud_learning_refresh.log
```

典型命令结构：

```bash
python scripts/run_automation_center.py \
  --mode mixed \
  --league 世界杯 \
  --historical-dates 2 \
  --historical-lookback-days 120 \
  --workers 1 \
  --task-timeout 480 \
  --max-events 3 \
  --max-analysis 12 \
  --max-intelligence 4 \
  --max-validation-dates 2 \
  --force-learning \
  --apply
```

设计原则：

- 低频，但更深。
- 允许跑学习、回测、未来比赛重分析。
- 不放在 15 分钟 tick 里，避免 2GB 云服务器内存被打满。
- 不能因为夜间学习导致白天页面打不开。

## 4. 当前已经解决的问题

### 4.1 页面空白 / Internal Server Error

问题：

- 云端前端拿 `/api/v1/lottery/matches` 时遇到 HTTP 500。
- 前端把 `"Internal Server Error"` 当 JSON 解析，报 `Unexpected token 'I'`。
- 后端健康检查曾出现 `database is locked`。

处理：

- 同步了新的前端构建。
- 后端接口恢复。
- 前端 fetch 增加非 JSON/HTTP 错误保护。
- 云端 API 使用外部 timer 替代进程内 scheduler。

当前：

```bash
curl -s http://localhost:8000/api/v1/health
```

返回：

```json
{"status":"healthy","database":"connected","matches_count":382565}
```

### 4.2 旧 scheduler 锁库

问题：

- 旧 `daily_runner` cron 和 API/自动化抢 SQLite。
- 日志里出现过 `sqlite3.OperationalError: database is locked`。

处理：

- 云端 `DISABLE_DAILY_SCHEDULER=1`。
- 删除 root crontab 的旧 `daily_runner`。
- 主库和 oddsfe 库开启 WAL。
- 任务改为 systemd timer 独立进程。

### 4.3 自动化可见性

问题：

- 前端原来只知道 APScheduler，不知道 systemd timer。
- API 进程内 scheduler 禁用后，前端容易显示“调度器未运行”。

处理：

- `/api/scheduler/status` 增加外部 timer 状态。
- 返回 `external_automation`、`external_learning`、`external_timers`。
- `jobs` 里加入 `learning_refresh`。

### 4.4 云端磁盘

问题：

- `/opt/football_backups` 历史备份占约 12G。
- 系统盘一度到 78%。

处理：

- 生成清理清单：

```text
/opt/football_tools_backups/cleanup_manifests/football_backups_before_cleanup_20260626_172459.txt
```

- 清理 `/opt/football_backups` 子内容。
- 保留目录本身。
- 系统盘从 78% 降到 53%。

当前原则：

- 云端不要长期堆大备份。
- 运行时短保留备份放：

```bash
/opt/football_tools_backups/runtime
```

- 本地或单独备份盘才是长期备份位置。

## 5. 未来目标状态

### 5.1 一句话目标

最终要形成一个 24 小时运行的足球数据与分析系统：

白天持续采集今日/明日比赛，赛中和赛后自动补比分、半场、赛果、O/U，赛后自动复盘，空闲时倒序补历史，夜间低频学习和回测，学习后自动重分析未开赛比赛，前端能看到自动化状态、数据质量、命中率变化和缺口处理进度。

### 5.2 不是要做什么

不是：

- 单场拍脑袋。
- 只拿赔率胡乱推。
- 已结束比赛反复用答案倒推。
- 没数据也硬说“分析过”。
- 所有任务串行跑到卡死。
- 页面打不开还在后台疯狂写库。

### 5.3 要做什么

要做：

1. 真实采集。
2. 原始证据留存。
3. 数据质量评估。
4. 分析前缺口判断。
5. 缺什么自动采什么。
6. 每场比赛生成分析快照。
7. 赛后验证每个玩法。
8. 错误归因。
9. 相似案例沉淀。
10. 模型/规则改动有回测。
11. 学习后重分析未开赛比赛。
12. 前端能看到自动化运行状态。

### 5.4 理想自动化流程

```text
比赛进入视野
  ↓
采集 Sporttery / oddsfe / WorldCup context
  ↓
检查缺口
  ↓
缺赔率 -> sporttery/oddsfe
缺 O/U -> oddsfe_ou_line
缺赛果 -> oddsfe_event_detail / sporttery result
缺半场 -> oddsfe_event_detail
缺情报 -> intelligence pipeline
缺阵容 -> API-Football / API-Sports / bifen188 / football-data.org squad
缺伤停 -> news/API/injury sources
缺天气 -> wttr.in / weather fetcher
缺近期攻防 -> team_match_facts
  ↓
生成分析
  ↓
保存 feature/context snapshot
  ↓
比赛结束
  ↓
补赛果/半场/O-U 结果
  ↓
验证 SPF/RQSPF/BQC/O-U/BF
  ↓
错误归因
  ↓
写 post_match_reviews / similar_match_cases
  ↓
夜间学习/回测
  ↓
通过 gate 才应用
  ↓
重新分析未开赛比赛
```

### 5.5 分段并发目标

任务不能一条线跑死。

目标结构：

```text
中控 Orchestrator
  ├─ 采集任务池
  ├─ 情报任务池
  ├─ 赔率/O-U 任务池
  ├─ 赛果补齐任务池
  ├─ 分析任务池
  ├─ 复盘任务池
  ├─ 学习任务池
  └─ 健康检查/熔断
```

要求：

- 每个任务有 timeout。
- 每个任务有 lock。
- 每个任务有 run_id。
- 每个任务能独立失败。
- 一个任务卡死不能堵住全部。
- 高频任务轻量，低频任务深度。
- API 不能被后台任务拖死。

### 6.3 代码同步方式

当前推荐脚本：

```powershell
.\deploy\sync_code_to_cloud.ps1
```

常用参数：

```powershell
# 正常同步，会构建前端，会重启服务
.\deploy\sync_code_to_cloud.ps1

# 不重新构建前端
.\deploy\sync_code_to_cloud.ps1 -SkipBuild

# 同步但不重启 API
.\deploy\sync_code_to_cloud.ps1 -NoRestart

# 应用快照清理
.\deploy\sync_code_to_cloud.ps1 -ApplySnapshotCleanup
```

默认行为：

1. 本地执行 `npm run build`。
2. 打包同步清单里的代码和脚本。
3. 排除数据库、CSV、备份文件。
4. 上传到云端 `/tmp`。
5. 解压到 `/opt/football_tools`。
6. 执行：

```bash
sudo bash deploy/cloud_post_sync_maintenance.sh
```

同步脚本默认排除：

```text
*.db
*.sqlite
*.csv
*.bak
*.backup
*_backup.*
__pycache__
*.pyc
```

原因：

- 防止误覆盖云端正在运行的主库。
- 防止把本地临时文件、备份包、CSV 原始数据塞进云端项目。

### 6.4 post-sync 维护做什么

文件：

```text
deploy/cloud_post_sync_maintenance.sh
```

它会做：

1. 修正数据目录权限。
2. 修正主库、oddsfe 库权限。
3. 设置运行时备份目录。

### 6.5 数据库同步原则

数据库不随代码同步。

原因：

- 主库 2.6G，oddsfe 库 2.2G。
- 云端正在写库，直接覆盖风险极高。
- 本地和云端可能都有新数据，简单覆盖会丢数据。
- 自动采集应该让云端自己补齐，不是频繁传大库。

数据库同步只有三种合理情况：

#### 情况 A：本地结构升级，需要迁移脚本上云

做法：

1. 同步代码和迁移脚本。
2. 云端备份或确认可回滚。
3. 云端执行 migration。
4. 检查表结构和行数。
5. 检查 API。

不要直接传整个 DB。

#### 情况 B：本地数据库明显比云端完整，需要一次性导入

做法：

1. 停掉云端自动化 timer。
2. 停掉 API 或至少确认只读。
3. 备份云端当前 DB 到本地。
4. 上传本地 DB 到临时路径。
5. 校验大小、表数、关键行数。
6. 替换。
7. 修权限。
8. 开 WAL。
9. 重启服务。
10. 跑 API 健康检查。

参考命令：

```bash
sudo systemctl stop football-automation-tick.timer football-learning-refresh.timer
sudo systemctl stop football-analyst.service

cp /opt/football_tools/data/football_v2.db /opt/football_tools_backups/runtime/football_v2_before_replace.db

# 上传新库到 /tmp/football_v2.db 后：
sudo mv /tmp/football_v2.db /opt/football_tools/data/football_v2.db
sudo chown ubuntu:ubuntu /opt/football_tools/data/football_v2.db
sudo chmod 664 /opt/football_tools/data/football_v2.db

sqlite3 /opt/football_tools/data/football_v2.db "PRAGMA journal_mode=WAL;"

sudo systemctl start football-analyst.service
sudo systemctl start football-automation-tick.timer football-learning-refresh.timer
```

#### 情况 C：云端数据更完整，需要拉回本地

做法：

```powershell
scp -i $HOME\.ssh\football_server `
  ubuntu@1.117.70.20:/opt/football_tools/data/football_v2.db `
  D:\football_tools\_server_backups\football_v2_cloud_YYYYMMDD_HHMMSS.db
```

建议：

- 拉回本地先放 `_server_backups`。
- 不要直接覆盖本地 `data/football_v2.db`。
- 对比表数、行数、最近更新时间后再决定是否替换。

## 9. 分析逻辑未来要达到的标准

### 9.1 分析不是孤立玩法

胜平负、让球胜平负、半全场、大小球、比分不是互相独立的。

它们应该形成同一个底层判断：

```text
胜负方向
  ↓
净胜球区间
  ↓
总进球区间
  ↓
双方是否进球
  ↓
半场节奏
  ↓
比分候选
  ↓
SPF / RQSPF / BQC / O-U 一致性检查
```

举例：

- 如果主胜很强，且主队让 -1，不能随便给出和比分、半全场冲突的 RQSPF。
- 如果预测总进球 3.2，真实 O/U 盘是 2.5，却给小球，需要非常明确的降速理由。
- 如果半全场给负负，全场却判断主胜，那必须被一致性 gate 拦住。

### 9.2 大小球不是只看盘口

需要综合：

- 主队近期进球。
- 主队近期丢球。
- 客队近期进球。
- 客队近期丢球。
- 双方进球概率。
- 强弱差距。
- 是否强队稳固防守。
- 是否弱队有反击能力。
- 盘口是 2、2.25、2.5、2.75、3、3.25 还是 3.5。
- 当前小组形势。
- 是否末轮。
- 是否必须抢净胜球。
- 天气。
- 阵容。
- 伤停。
- 赛程疲劳。
- 历史相似盘。

### 9.3 比分不是装饰

比分候选应该服务于玩法一致性。

例如：

- 主胜 + 大 2.5 + 客队有进球概率高：常见候选可能是 2-1、3-1。
- 主胜 + 小 2.5 + 客队进球概率低：常见候选可能是 1-0、2-0。
- 客胜 + 受让方向 + 低节奏：常见候选可能是 0-1、1-1、0-0。

比分预测不一定命中，但它必须解释玩法之间是否连贯。

### 9.4 半全场的底层

半全场不是随便拼两个字。

要看：

- 两队上半场进球率。
- 两队上半场丢球率。
- 是否慢热。
- 是否强队早压迫。
- 是否弱队先守。
- 是否末轮有策略。
- 半场平局概率。
- 全场方向。

半全场至少要满足：

```text
半场方向 + 全场方向 = 和 SPF/RQSPF/比分候选不冲突
```

### 9.5 让球胜平负的底层

让球胜平负本质是净胜球分布。

主队让 -1：

| 实际净胜 | RQSPF |
|---|---|
| 主胜 2 球或更多 | 让胜 |
| 主胜 1 球 | 让平 |
| 平/负 | 让负 |

主队受让 +1：

| 实际结果 | RQSPF |
|---|---|
| 主胜或平 | 让胜 |
| 主负 1 球 | 让平 |
| 主负 2 球或更多 | 让负 |

所以模型必须先判断：

```text
方向 + 净胜球概率 + 盘口线
```

不能只根据赔率最小值随便选。
