# 采集能力与数据底座梳理

> 更新时间: 2026-06-18  
> 目标: 梳理当前项目已有采集能力、数据库资产、好思路、问题和下一步底座改造方向。

---

## 1. 总体判断

当前项目不是缺少采集能力，而是能力已经很多，但分散在多条链路里。

现在最强、最稳定、最接近实战闭环的是:

```text
sporttery 体彩赛事入口
  -> oddsfe 赔率 / 赛果 / event_id 桥接
  -> lottery_matches / lottery_odds / lottery_results
  -> analyze / validate / learn
  -> 体彩中心前端展示
```

云端已经能跑出较完整的 daily loop:

- 今日体彩赛事有赔率覆盖。
- oddsfe 和 sporttery 均为 healthy，其中 oddsfe 成功率更高。
- 已经有赛后验证和归因。
- 已经能提前采集未来比赛并生成分析。

但距离最终目标还差一个关键层:

```text
统一采集任务
统一原始证据留存
统一数据质量判断
统一赛前快照
统一赛后复盘
统一相似比赛检索
```

也就是说，现在有“数据”和“分析”，但还没有完全形成“可复盘、可训练、可扩展、可追踪”的数据工厂。

## 2. 当前已有能力

### 2.1 体彩赛事与玩法赔率

核心位置:

- `backend/app/lottery/services/sync_service.py`
- `backend/app/lottery/data_sources/scrapers/lottery_crawler.py`
- `backend/app/lottery/routers/lottery.py`
- `backend/app/lottery/dao/lottery_dao.py`

已有能力:

- 拉取体彩开售比赛。
- 保存 SPF / RQSPF / BF / BQC / TTG 等玩法赔率。
- 从 `rqspf.goal_line` 修复让球方向。
- 通过 `lottery_match_id` 串联赛事、赔率、结果、分析报告。
- 前端体彩中心能展示每日比赛卡片、推荐、赔率、状态和赛果。

好处:

- 体彩中心适合作为主枢纽。
- 玩法结构天然贴合用户实际关注点。
- 比赛结束后可以直接验证 SPF、让球、半全场、比分、大小球。

问题:

- `lottery_odds` 里 opening/current/latest/closing 的含义还不够统一。
- 部分接口里 `latest` 为空，说明临场赔率轨迹仍不足。
- 玩法复盘样本偏向 SPF，RQSPF/BF/BQC 样本明显少。

### 2.2 oddsfe 赔率与赛果主源

核心位置:

- `fetchers/odds_feed_api/`
- `scripts/oddsfe_sync.py`
- `backend/app/core/collect.py`
- `backend/app/core/validate.py`
- `scripts/sync_finished_results.py`

已有能力:

- 拉取 oddsfe schedule。
- 桥接 oddsfe event_id。
- 获取 Pinnacle 1X2。
- 获取 O/U 盘口。
- 通过 event API 获取 score_details。
- 补半场比分、全场比分、BQC、BF、RQSPF。
- 将 oddsfe 历史比赛同步进 `matches`。

好处:

- 云端验证显示 oddsfe 是当前最健康的数据源。
- oddsfe 的 event_id 是体彩赛事与海外赔率/赛果之间的重要桥。
- 赔率结构非常适合后续做相似比赛检索。

问题:

- auth 过期、接口变更仍是风险。
- `oddsfe_merged.db` 与 `football_v2.db` 的边界需要更清楚。
- 当前更重视“拿到当前赔率”，还没有完整的盘口时间序列。

### 2.3 日循环

核心位置:

- `backend/app/core/daily_runner.py`
- `backend/app/core/collect.py`
- `backend/app/core/analyze.py`
- `backend/app/core/validate.py`
- `backend/app/core/learn.py`

当前流程:

```text
perceive -> collect -> intel -> classify -> analyze -> push -> clv_update -> validate -> learn
```

已有能力:

- 自动恢复卡住的 daily cycle。
- 采集今日/明日窗口。
- 分析体彩比赛和近期国家队比赛。
- 赛后同步结果并验证。
- 根据验证结果做参数学习。

好处:

- 已经具备自动化骨架。
- 后续扩展采集模块时，可以挂到这个流程。

问题:

- 各节点输出更多是 JSON 结果，不足以作为长期训练样本。
- 采集运行过程缺少统一的 `collection_runs` 和 `source_artifacts`。
- 失败原因、降级、原始响应、字段缺失没有统一结构。

### 2.4 情报中心

核心位置:

- `backend/app/intelligence/service.py`
- `backend/app/intelligence/external_collectors.py`
- `backend/app/intelligence/storage.py`

已有能力:

- 创建 intelligence job。
- 按赛事类型识别分析视角。
- 生成 requirement 模板。
- 采集天气、球队新闻、伤停、预计阵容。
- 生成 intelligence package。
- 存储 intelligence reviews。

好处:

- 这部分非常接近用户想要的“每场比赛需要哪些数据、缺什么就补什么”。
- requirement 模板已经区分世界杯/国家队/联赛。
- 天气、伤停、新闻、阵容都能作为内容层证据。

问题:

- 情报中心还没有完全成为 daily loop 的强制环节。
- 外部采集结果更像“包”，还没有沉淀成统一赛前快照。
- 与体彩中心的展示和分析模型耦合还不够强。

### 2.5 世界杯 2026 上下文

核心位置:

- `backend/app/worldcup/service.py`
- `backend/app/worldcup/routes.py`
- `frontend/src/components/WorldCupContext.vue`

核心位置:

- `backend/app/data_sources/source_knowledge.py`
- `backend/app/pipeline/collector.py`

已有能力:

- 需要先把 oddsfe、sporttery 这两条最强链路纳入统一 source registry，再扩其他源。

## 3. 数据资产判断

### 3.1 本地数据库

本地 `data/football_v2.db` 已确认:

- 大小约 280 MB。
- 约 85 张表。
- `matches` 约 137,858 场。
- `teams` 约 18,420 队。
- `players` 约 26,262 人。
- `match_odds` 约 18,467 条。
- `match_lineups` 约 3,731 条。
- `statsbomb_passes` 约 582,417 条。
- `lottery_analysis_reports` 约 549 条。
- `intelligence_packages` 约 49 个。

本地价值:

- 适合做开发和迁移验证。
- 已经有足够结构验证数据底座设计。

### 3.2 云端数据库

云端公开接口确认:

- 主库约 1.4 GB。
- `/api/v1/health` 显示 `matches_count=382532`。
- oddsfe + sporttery 当前健康。
- 当前体彩比赛赔率覆盖完整。
- 赛果回填状态较好，但仍有少量缺口。

云端价值:

- 更接近生产数据资产。
- 已经沉淀大量 oddsfe 历史比赛。
- 不应直接覆盖或删除。

### 3.3 资产结论

当前最值钱的数据不是单一 odds，也不是单一新闻，而是:

```text
比赛实体 + 赔率结构 + 赛前内容 + 赛制上下文 + 赛后结果 + 复盘归因
```

现在项目已经有前两项和部分结果验证，下一步要补齐“赛前内容快照”和“赛后复盘归因”。

## 5. 主要问题

### 5.1 采集入口太多

现在同时存在:

- core daily loop。
- lottery sync service。
- scripts 下的 oddsfe/sporttery 手工脚本。
- pipeline collector/loader。
- data_sources manager。
- intelligence collectors。

问题不是多，而是缺少统一编排:

```text
这场比赛需要哪些数据
先找哪个源
失败后找哪个源
采到了什么原始证据
字段是否可信
是否可以进入分析
```

### 5.2 原始证据留存不足

现在很多采集结果直接写业务表。这样短期好用，但长期复盘时会问:

- 当时接口原始返回是什么？
- 当时数据是不是旧的？
- 当时哪个字段缺失？
- 是哪个源给出的？
- 源之间是否冲突？

这些需要 `source_artifacts` 或类似结构。

### 5.3 odds 快照还不够稳定

当前有 opening/current/closing/midday 的影子，但还没完全标准化。

需要明确:

- opening: 第一次有效采集。
- current: 当前展示用最新值。
- midday: 中午/固定时间点快照。
- closing: 临场或封盘前最后有效值。
- latest: 可与 current 合并或明确为最后抓取值。

没有这个，CLV、赔率异动、临场信息差都不稳定。

### 5.4 玩法复盘样本不均衡

云端当前:

- SPF 样本最多。
- OU 有一些。
- RQSPF/BF/BQC 样本很少。

体彩中心如果要真正强，不能只验证主胜平负。

### 5.5 世界杯上下文未进入主分析

世界杯模块不能只是一个展示页，还应进入单场分析:

- 当前第几轮小组赛。
- 前两轮结果。
- 本场是否必须赢。
- 平局是否可接受。
- 是否要追净胜球。
- 是否可能为半区选择保守/轮换。
- 同组另一场同时比赛的影响。

### 5.6 云端与本地版本差异

云端主库更大，功能更接近生产；本地代码更新更快。

风险:

- 本地有 World Cup 2026 接口，云端没有。
- 云端有更多数据，本地不一定覆盖。
- 若直接同步数据库，可能覆盖云端资产。

## 6. 需要补齐的底座

### 6.1 `collection_runs`

记录每次采集任务:

- `run_id`
- `trigger_source`
- `run_type`
- `match_date`
- `status`
- `started_at`
- `finished_at`
- `summary_json`
- `error`

意义:

- 以后能知道哪一天采集跑没跑。
- 能追踪失败源和失败频率。

### 6.2 `source_artifacts`

记录原始证据:

- `artifact_id`
- `run_id`
- `source_name`
- `source_type`
- `entity_type`
- `entity_id`
- `payload_json`
- `payload_hash`
- `confidence`
- `captured_at`

意义:

- 保留原始接口/网页/解析结果。
- 支持复盘“当时我们看到了什么”。

### 6.3 `source_entity_mappings`

统一映射:

- 体彩队名。
- oddsfe team id。
- football-data.org team id。
- FIFA code。
- 中文名/英文名/别名。

意义:

- 国家队和小联赛队名问题会越来越少。
- 解决 DR Congo/Congo DR、Korea Republic/South Korea 这类问题。

### 6.4 `match_context_snapshots`

记录赛前上下文:

- 比赛基础信息。
- 世界杯/联赛/杯赛视角。
- 小组形势。
- 赛程压力。
- 天气。
- 伤停。
- 新闻。
- 阵容预测。
- 数据完整度。

意义:

- 比赛结束后仍能回看赛前信息。

### 6.5 `match_feature_snapshots`

记录模型特征:

- Elo。
- Poisson。
- 盘口。
- 赔率隐含概率。
- 近期状态。
- 主客/中立场。
- 动机。
- 风险项。

意义:

- 训练模型和相似比赛检索都依赖它。

### 6.6 `post_match_reviews`

记录赛后复盘:

- 预测结果。
- 实际结果。
- 命中/未命中。
- 正确因素。
- 漏掉因素。
- 市场是否正确。
- 模型是否逆市场。
- 调整建议。

意义:

- 不只是知道错了，而是知道为什么错。

### 6.7 `similar_match_cases`

记录相似比赛:

- 当前比赛。
- 历史相似比赛。
- 相似维度。
- 相似分数。
- 历史结果。
- 参考结论。

意义:

- 实现用户想要的“类似盘口、类似赔率、类似实力再出现时，提高判断准确性”。

## 8. 优先级路线

### P0: 保住当前可用链路

- 不动云端主库。
- 本地继续验证。
- 保持 sporttery + oddsfe + lottery center 正常。
- 世界杯接口同步上云前先备份。

### P1: odds 快照标准化

- 统一 opening/current/latest/closing/midday 定义。
- 确保每次采集不覆盖 opening。
- 临场前写入 closing。
- 建立 odds movement。
- 体彩中心显示赔率变化。

### P2: 复盘样本补全

- 对已结束比赛补 BF/BQC/RQSPF/OU。
- `lottery_validation` 按玩法写入。
- 前端可按玩法查看命中率。

### P3: 赛前上下文快照

- 从 intelligence package 生成 match context snapshot。
- 把世界杯小组形势写入单场上下文。
- 每场比赛分析前固定保存“当时信息”。

### P4: 相似比赛检索

先规则化，不急着上复杂模型:

- 赔率区间相似。
- 盘口相似。
- 大小球线相似。
- 强弱差相似。
- 赛事类型相似。
- 小组/淘汰/联赛动机相似。
- 阵容伤停相似。

### P5: 模型训练与 Agent 接入

- 有足够赛前快照和赛后复盘后，再做模型训练。
- Agent 用于归因和建议，不直接覆盖核心数据。

## 9. 近期具体执行建议

1. 先运行只读检查脚本，确认本地和云端差异。
2. 同步本地 World Cup 2026 接口到云端。
3. 重启云端后验证 `/api/v1/world-cup/2026/health`。
4. 补 `collection_runs` 和 `source_artifacts` 的迁移脚本，但先 dry-run。
5. 先不改大模型，优先把 odds 快照和复盘闭环做稳。
6. 把相似比赛检索先做成 SQL/rule 版本，接到单场详情页。

---

## 10. 结论

这个项目现在最强的是 odds 主轴，尤其是 oddsfe。

但最终要做强，不能只靠 odds。正确方向是:

```text
odds 负责量化骨架
数据内容负责证据解释
世界杯/联赛上下文负责场景判断
赛后复盘负责持续学习
相似比赛负责经验复用
```

现在要做的不是重写项目，而是把已有能力收束成一个稳定的数据底座。
