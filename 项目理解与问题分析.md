# 项目理解与问题分析

> 分析时间：2026-06-18  
> 分析范围：前端、后端、架构、数据采集、数据存储、现有文档与本地 SQLite 数据库。  
> 说明：本报告只做理解和问题梳理，没有修改业务代码，也没有做完整联调压测。

## 一、项目概览

这个项目是一个足球数据采集、清洗、分析和竞彩预测平台。它不是单纯的比分展示站，而是想把多来源足球数据整理成可查询、可分析、可预测的业务系统。

从代码和文档看，项目主要有几条线：

- 前端展示线：Vue + Vite，提供首页、赛事前瞻、分析中心、数据中心、体彩中心、球队/联赛/比赛详情等界面。
- 后端 API 线：FastAPI，提供比赛、球队、联赛、排名、同步、数据源、分析、竞彩、AI 分析等接口。
- 数据采集线：`fetchers/`、`backend/app/data_sources/`、`backend/app/pipeline/`、`backend/scripts/` 中都有采集、同步、补全、导入脚本。
- 分析预测线：`backend/app/analytics/`、`backend/app/core/`、`backend/app/lottery/` 中有 Elo、xG、Poisson、赔率、动机、疲劳、天气、价值投注、复盘、模型学习等逻辑。
- 数据存储线：目前并存 `data/football_v2.db`、`data/unified_football.db`、`fetchers/odds_feed_api/oddsfe_merged.db`，同时还有大量 CSV、JSON、映射文件和备份文件。

我对本地数据库做了只读检查，当前能看到的核心规模大致是：

- `data/football_v2.db`：约 275.58 MB，79 张表，`matches` 表 137,858 场比赛。
- `data/unified_football.db`：约 253.12 MB，9 张表，`match_data` 371,641 行；普通只读读取表结构时报 `unable to open database file`，用 immutable 只读方式可以打开。
- `fetchers/odds_feed_api/oddsfe_merged.db`：约 2.27 GB，单表 `oddsfe`，250,217 行，378 列。
- 竞彩相关：`lottery_matches` 126 行，`lottery_odds` 235 行，`lottery_results` 45 行，`lottery_predictions` 378 行。

整体判断：项目已经积累了大量数据和功能，方向是有价值的；但当前更像“快速扩展出来的多功能工作台”，还没有完全收敛成一个边界清晰、数据链路稳定、前后端契约统一的产品化系统。

## 二、好的地方

### 1. 数据覆盖面很宽

项目不是只接一个 API，而是尝试整合 API-Football、API-Sports、football-data.co.uk、StatsBomb、TheSportsDB、OpenWeatherMap、新闻源、赔率源、世界杯/国家队专项数据等。对足球分析来说，这是很重要的基础能力。

尤其是赔率和历史比赛数据已经有明显积累：

- `oddsfe_merged.db` 有 25 万多行赔率相关数据。
- `football_v2.db` 的 `matches` 有 13 万多场比赛。
- 文档中也已经明确区分了训练数据、实时预测数据、赔率数据、xG 数据、FIFA 排名等用途。

### 2. 业务野心清楚，不只是 CRUD

后端已经覆盖很多分析模块：Elo、xG、Poisson、H2H、球队状态、主客场、动机、天气、裁判、场地、疲劳、联赛形势、赔率、新闻、实时比赛、价值投注、竞彩预测和复盘。

这说明项目的目标不是“展示数据”，而是希望形成“赛前数据收集、模型预测、赛果验证、参数迭代”的闭环。

### 3. 文档意识比较好

项目里有不少有价值的文档，例如：

- `PROJECT.md`：完整列出了项目结构、数据库、数据源、分析模块和已知问题。
- `docs/DATA_UPDATE_FLOW.md`：说明数据更新流程。
- `docs/PROJECT_DATA_SEPARATION.md`：讨论代码和数据分离。
- `backend/app/lottery/ARCHITECTURE.md`、`DATA_STORAGE_DESIGN.md`：说明竞彩模块设计。
- `docs/missing_features_analysis.md`、`docs/analysis_center_missing_data.md`：记录缺失能力。

这对后续重构很重要，因为项目上下文没有完全散落在代码里。

### 4. 已经有模块化的雏形

虽然目前边界还不够干净，但目录上已经有一些合理分层：

- `backend/app/routers/` 放基础业务路由。
- `backend/app/analytics/` 放分析能力。
- `backend/app/lottery/` 放竞彩专项能力。
- `backend/app/data_access/` 有 DAO 雏形。
- `backend/app/pipeline/` 和 `backend/app/data_sources/` 尝试管理采集任务。
- `fetchers/storage/` 已经有 schema、database、crud 这样的存储抽象尝试。

这些都说明项目不是完全无结构，只是结构之间还没有统一到一套主线。

### 5. 部署和运行入口已经存在

项目有 `Dockerfile`、`docker-compose.yml`、`nginx.conf`、`start.sh`、`start.bat`、`.env.template`，说明已经考虑过本地运行和部署。虽然还需要加强环境隔离和数据挂载策略，但基础入口已经具备。

## 三、不好的地方

### 1. 项目已经出现明显的“多套系统并存”

目前至少有三套并行思路：

- 旧主库和主业务：围绕 `football_v2.db`。
- 统一数据层：围绕 `unified_football.db` 和 `match_data` JSON。
- 赔率专项库：围绕 `oddsfe_merged.db` 宽表。

这三套系统各自都有价值，但现在缺少一个明确的主架构说明：谁是事实源，谁是缓存，谁是训练库，谁是业务查询库，谁负责同步，谁负责校验。

### 2. 功能增长快于边界治理

后端存在大文件和大路由堆叠：

- `backend/app/main.py` 仍然包含大量直接 API：数据统计、CSV 读写、缺失检测、数据修复、循环状态、回测、预测、健康检查等。
- `backend/app/analytics/routes.py` 超过 100 KB，包含大量分析接口。
- `backend/app/lottery/routers/lottery.py` 超过 60 KB，同时处理查询、分析、同步、回填、映射、健康检查、pipeline 等。

这会带来维护成本：改一个功能时很难判断影响范围，接口、数据访问、业务逻辑和运维任务容易混在一起。

### 3. 数据质量问题已经影响分析可信度

主库 `matches.status` 当前有多种表达方式：`Finished`、`finished`、`FINISHED`、`scheduled`、`Not Started`、`FT`、空字符串等。状态不统一会直接影响“已完赛/未赛/可验证/可预测”的判断。

本地只读检查还看到：

- `matches` 中比赛时间为空的记录有 17,185 条。
- `lottery_matches` 126 行中，有 50 行没有关联赔率，有 81 行没有关联赛果。
- `lottery_predictions` 当前检查到的 `play_type` 全部是 `spf`，其他玩法预测链路还没有实际覆盖或没有落库。

这说明模型和前端展示不一定错，但它们依赖的数据基础还不够稳。

### 4. 修复脚本和临时脚本过多

根目录和 `backend/scripts/` 中有不少 `tmp_*`、`fix_*`、`clean_*`、`check_*`、`fill_*`、`sync_*` 脚本。这些脚本对快速推进很有用，但如果没有沉淀成正式 pipeline，会导致：

- 不知道哪个脚本是当前标准流程。
- 同一个问题可能被多个脚本以不同方式修复。
- 新采集的数据可能绕过校验直接写入库。
- 线上和本地数据状态越来越难复现。

### 5. 数据文件和代码目录耦合重

项目把大体量数据库、CSV、JSON、备份、脚本、前后端代码放在一个工作区中。`PROJECT_DATA_SEPARATION.md` 已经意识到这个问题，但从当前目录看，数据和代码还没有完全分离。

长期看，这会影响 Git 管理、部署、备份、迁移和协作。

## 四、前端问题

### 1. 页面状态和路由系统并存

`frontend/src/App.vue` 使用 `currentPage` 控制主页面切换，例如首页、赛事前瞻、数据查看、分析中心、体彩中心等；同时 `frontend/src/main.js` 又定义了 Vue Router，用于 `/team/:id`、`/league/:id`、`/match/:id` 和多个 `/analysis/*` 页面。

这导致前端实际有两套导航系统：

- 菜单点击只改 `currentPage`。
- 详情页和分析子页走 Router。
- `selectPage()` 在切换菜单时会 `router.push('/')`，所以用户当前看到的页面不会体现在 URL 中。

影响：刷新、分享链接、浏览器前进后退、页面恢复状态都会比较弱。

### 2. 搜索跳转到未定义路由

`frontend/src/App.vue` 的搜索逻辑调用：

```js
router.push({ name: 'Analysis', query: { q: searchQuery.value } })
```

但 `frontend/src/main.js` 中没有名为 `Analysis` 的路由，只有 `HandicapAnalysis`、`EfficiencyAnalysis`、`MLPrediction` 等具体分析子路由。

影响：搜索功能可能直接跳转失败或出现路由警告，属于明确的前端 bug。

### 3. API 封装存在重复定义

`frontend/src/api/index.js` 中 `getMatchImportance` 定义了两次。后面的定义会覆盖前面的定义。

影响：调用方以为传了 `seasonId`，实际可能被后一个签名吞掉，造成参数不一致、接口结果不稳定或调试困难。

### 4. 页面加载自动触发后端同步

`frontend/src/App.vue` 在页面挂载时会请求 `/api/v1/sync/start`。这让“打开页面”与“启动数据同步”耦合在一起。

风险：

- 用户只是访问页面，也可能触发后台采集或写库。
- 多用户访问会重复触发同步。
- 前端刷新可能造成后端任务堆积。
- 同步失败会影响页面初始体验。

更合理的方式是把同步变成用户明确操作，或由后端调度器统一管理。

### 5. 前端模块很多，但统一状态管理不足

当前组件数量较多，业务面很宽，但从入口看主要依赖组件本地状态和 API 模块。随着体彩、分析中心、数据中心继续扩展，容易出现：

- 多组件重复请求同一数据。
- 同一字段在不同页面显示口径不一致。
- 加载、错误、空状态处理风格不统一。
- 前端不知道后端数据新鲜度和同步状态。

## 五、后端问题

### 1. `main.py` 承担了过多职责

`backend/app/main.py` 不只是应用启动文件，还直接包含很多接口和业务逻辑，例如：

- `/api/v1/data/stats`
- `/api/v1/csv/read`、`/api/v1/csv/write`
- `/api/v1/data/detect-missing`
- `/api/v1/data-fix/*`
- `/api/cycle/*`
- `/api/backtest`
- `/api/oddsfe-backtest`
- `/api/validation/*`
- `/api/analysis/{lottery_match_id}`
- `/api/warmup/*`

这会让 `main.py` 变成“所有临时功能的入口”，后续很难测试和拆分。

### 2. 分析路由过于集中

`backend/app/analytics/routes.py` 一个文件中包含大量不同领域的接口：Elo、xG、Poisson、H2H、天气、赔率、实时、新闻、TheSportsDB、时间转换、盘口、效率、教练、爆冷、赛季情景、追踪验证、agent 等。

问题不是功能多，而是多种领域共享同一个路由文件，会导致：

- import 依赖越来越复杂。
- 某个分析模块异常可能影响整组路由加载。
- 很难给不同模块做独立测试。
- API 文档虽然有 tag，但代码层面边界不清。

### 3. 数据访问方式不统一

项目里已经有 `backend/app/data_access/` DAO，但很多地方仍然直接 `sqlite3.connect()` 并拼路径。`rg` 能看到大量模块直接连接 `football_v2.db`，也有直接硬编码 `d:/football_tools/data/football_v2.db` 的情况。

风险：

- 数据库路径难统一配置。
- 测试环境难替换数据库。
- 连接参数、超时、只读/写入策略不统一。
- 事务和错误处理分散。

### 4. 业务路由直接依赖多个数据库

`backend/app/lottery/routers/lottery.py` 同时定义：

- `DB_PATH = ... data/football_v2.db`
- `ODDSFE_DB_PATH = ... fetchers/odds_feed_api/oddsfe_merged.db`

路由层直接知道两个数据库的位置，并直接查询它们。短期很方便，长期会让 API 层和存储布局强耦合。

更好的边界应该是：路由调用 service，service 调用 repository/DAO，DAO 负责具体库和表。

### 5. 相似逻辑重复出现

`backend/app/routers/matches.py` 中有时区转换、北京时间过滤、重复比赛去重等逻辑，并且在多个接口中重复处理。`main.py` 中也存在比赛查询和数据处理逻辑。

风险：

- 同一场比赛在“今日比赛”“日期范围”“未来比赛”中可能过滤口径不同。
- 时区偏移用静态字典和手工 offset，夏令时、跨国赛事、UTC/本地时间混合时容易出错。
- 去重规则分散，后续难验证。

### 6. 后台任务缺少统一任务模型

后端有 sync、pipeline、scheduler、lottery pipeline、backfill、warmup、validate、daily cycle 等多个任务入口。它们看起来都能触发数据变更或分析，但缺少统一的任务状态、幂等键、重试、取消、日志追踪模型。

影响：出了问题以后，很难回答：这次任务是谁触发的、处理了哪些日期、写了多少行、失败在哪一步、是否可以安全重跑。

## 六、架构问题

### 1. 主链路不够清晰

从项目目标看，理想主链路应该类似：

```text
data source -> 标准化 -> 实体匹配 -> 数据质量校验 -> 入库 -> 特征生成 -> 预测 -> 赛果回填 -> 复盘学习 -> 前端展示
```

当前这些环节都存在，但散落在多个目录和脚本中：

- `fetchers/` 有采集和 storage。
- `backend/app/data_sources/` 有数据源管理。
- `backend/app/pipeline/` 有任务和 schema 查询。
- `backend/scripts/` 有大量导入、同步、补全、清理脚本。
- `backend/app/core/validate.py` 又承担验证和回填。
- `backend/app/lottery/services/sync_service.py` 有竞彩同步。

问题是“能力存在”，但“唯一标准流程”不够明确。

### 2. 新旧架构没有完成收敛

`football_v2.db` 是当前业务主库，`unified_football.db` 像是新一代统一数据层，`oddsfe_merged.db` 是赔率训练/历史数据核心库。但现在三者并存，且代码直接跨库访问。

需要明确：

- 线上 API 查询以哪个库为准。
- 训练和回测以哪个库为准。
- 赔率宽表是否只作为原始层，还是直接参与业务查询。
- `unified_football.db` 是继续推进，还是作为实验性存储冻结。
- 多库之间的数据同步由谁负责，如何校验一致性。

### 3. 缺少清楚的领域边界

当前至少可以拆出几个领域：

- 基础足球资料：球队、联赛、赛季、比赛、排名、积分榜。
- 数据采集与源管理：API、爬虫、CSV、JSON、手工映射。
- 分析模型：Elo、xG、Poisson、疲劳、动机、天气、赔率等。
- 竞彩业务：赛事、玩法、赔率、预测、赛果、复盘。
- 用户功能：收藏、设置、可见联赛。
- 运维任务：同步、补全、回填、清理、备份、健康检查。

现在这些领域在路由、脚本和数据库中交叉较多，导致后续功能越多，系统越难稳定。

### 4. 配置管理不集中

项目中能看到多种路径写法：相对路径、`Path(__file__)` 推导、硬编码 Windows 绝对路径、环境变量、配置文件。数据源 key、数据库路径、任务参数、采集频率、联赛白名单等应该统一放到配置层。

配置不集中会影响部署和测试，尤其是 Windows 本地、Linux 服务器、Docker 三种环境同时存在时。

### 5. 测试和质量闸门不足

项目里有一些测试脚本和检查脚本，但更像临时验证，不像稳定的自动化测试体系。

建议至少建立几类质量闸门：

- API contract 测试：前端调用的接口必须存在，参数必须匹配。
- 数据质量测试：状态枚举、主键唯一、赔率 JSON 可解析、赛果格式合法。
- pipeline 幂等测试：同一天采集重复跑不会制造重复数据。
- 预测链路测试：赛前数据不能泄露赛果，验证逻辑能稳定回填。
- 数据库迁移测试：schema 变更可重复执行，不依赖手工热修。

## 七、数据采集问题

### 1. 采集入口太多

当前采集相关入口分布在：

- `fetchers/`
- `fetchers/orchestrator/`
- `backend/app/data_sources/`
- `backend/app/pipeline/`
- `backend/scripts/`
- `backend/app/services/sync_service.py`
- `backend/app/services/auto_sync_service.py`
- `backend/app/lottery/services/sync_service.py`
- `backend/app/core/validate.py`

这些入口可能都能拉取或写入数据，但缺少一个统一调度中心和统一任务记录表。

### 2. 数据源标准化压力很大

项目同时处理 API、爬虫、CSV、JSON、手工映射。不同来源的字段命名、球队 ID、联赛 ID、时间格式、赔率格式、状态值都不一样。

目前已有 `data/linkage/` 下的大量映射 JSON 和报告，例如球队名映射、球队 ID、中文名、国家队列表、联赛中文名等。这是优点，也是维护压力信号。

风险：只要映射错了，后续 form、H2H、赔率、赛果、竞彩匹配都会跟着错。

### 3. 手工补丁文化较重

从脚本命名可以看到不少修复和补齐动作，例如 `fix_*`、`fill_*`、`clean_*`、`tmp_*`、`check_*`。这说明项目在持续处理真实数据问题，但也说明数据质量保障还没有前置到采集和入库阶段。

更健康的方式是：

- 采集后先进入 raw/staging 层。
- 标准化和实体匹配产生明确日志。
- 校验通过后再进入业务表。
- 修复动作沉淀成可重复 migration 或 pipeline step。

### 4. 增量采集和幂等性需要加强

目前可以看到近期同步、未来比赛同步、按日期回填、竞彩同步、结果补齐等能力，但缺少统一的“采集批次”和“幂等键”设计。

应该能回答这些问题：

- 某个数据源某一天采集了几次？
- 每次返回多少条，写入多少条，跳过多少条，失败多少条？
- 重跑会覆盖、追加，还是跳过？
- 哪些字段来自哪个源，更新时间是什么？
- 某条比赛数据是否被后续来源覆盖过？

### 5. API 限流、失败重试和源健康需要产品化

文档中已经记录了一些 API key、免费额度和数据源健康信息，数据库也有 `data_source_health` 这类表。但从代码分布看，源健康、限流、失败重试、降级策略还没有成为统一基础设施。

后续如果采集稳定性要上去，需要统一处理：

- 每个源的速率限制。
- 请求失败重试和退避。
- 采集失败告警。
- 字段覆盖率统计。
- 数据源优先级和降级策略。

## 八、数据存储问题

### 1. 多数据库角色不清

当前至少有三个重要 SQLite 文件：

- `football_v2.db`：主业务库。
- `unified_football.db`：统一数据实验库或新数据层。
- `oddsfe_merged.db`：赔率历史宽表库。

问题不是多库本身，而是缺少明确分层：raw、staging、normalized、mart、model training、serving。现在业务代码会直接跨库访问，导致存储结构一变，API 和分析逻辑都要跟着变。

### 2. SQLite 已经承担了偏重的生产职责

SQLite 很适合本地开发和轻量应用，但当前项目的数据规模和任务类型已经比较重：

- 赔率库单文件约 2.27 GB。
- 主库 79 张表。
- 有后台同步、回填、预测、复盘、前端查询并发访问。
- 有大宽表、JSON 字段、历史 CSV、模型训练数据。

如果继续扩大，SQLite 会在并发写入、长事务、锁、迁移、观测性方面遇到瓶颈。短期可以继续用，但要严格控制写入任务和 API 查询的边界；中长期建议评估 PostgreSQL 或 DuckDB/Parquet 作为分析层。

### 3. schema 约束不足

`lottery_results` 当前建表语句中 `result_id INT` 没有主键，`lottery_match_id` 只有唯一索引 `idx_lr_match_id`。代码中存在 `INSERT OR REPLACE` 的使用倾向。

风险：

- 表结构表达不清楚，不知道真正业务主键是 `result_id` 还是 `lottery_match_id`。
- `INSERT OR REPLACE` 在 SQLite 中实际可能删除旧行再插入新行，容易影响关联字段和创建时间。
- 如果唯一索引、外键、写入逻辑不一致，重复和覆盖问题会很隐蔽。

### 4. 状态、时间、球队实体需要统一规范

本地检查发现 `matches.status` 同时存在多种大小写和语义相近值。比赛时间也有大量缺失。时区转换逻辑在后端代码中以手工 offset 处理。

建议建立统一规范：

- 比赛状态枚举：scheduled、live、finished、postponed、cancelled、abandoned 等。
- 原始状态保留在 raw 字段，业务状态写入 normalized 字段。
- 比赛时间同时保留 source local time、UTC time、Beijing time、timezone/source_time_type。
- 球队实体必须通过统一映射表解析，保留 source_team_id 和 canonical_team_id。

### 5. JSON 和宽表的查询边界需要明确

`unified_football.db` 的 `match_data` 使用 JSON 存储多源数据，这种设计灵活，但不适合所有查询。`oddsfe_merged.db` 的 `oddsfe` 是 378 列宽表，也不适合直接作为业务 API 的主要查询模型。

建议明确：

- JSON 表适合作 raw/staging 或审计层，不适合作高频业务查询。
- 宽表适合离线训练、回测、特征抽取，不适合直接暴露给前端。
- 前端查询应该走整理后的 serving tables 或 materialized views。
- 模型训练应使用可版本化的数据集快照，避免训练数据随业务库写入悄悄变化。

### 6. 数据备份和迁移机制需要正规化

项目里有 `backups/`、`data_backups/`、多个 `.db` 文件和临时迁移脚本。备份意识是好的，但需要进一步正规化：

- 备份命名规则统一。
- 每次 schema 变化有 migration 文件。
- migration 可重复执行，可回滚或至少可校验。
- 备份与当前代码版本能对应。
- 大数据文件不要直接跟代码仓库生命周期绑定。

## 九、风险优先级

### P0：会直接影响功能正确性的点

1. 前端搜索跳转到不存在的 `Analysis` 路由。
2. 前端 `getMatchImportance` 重复定义，可能导致参数签名和真实请求不一致。
3. `matches.status` 状态值不统一，影响完赛、赛前、验证、回测判断。
4. 页面加载自动触发 `/api/v1/sync/start`，可能造成非预期写库和任务重复。
5. 竞彩数据链路不完整：126 场竞彩比赛中 50 场缺赔率、81 场缺赛果。

### P1：会持续拖慢开发和排错的点

1. `main.py`、`analytics/routes.py`、`lottery.py` 职责过重。
2. 后端直接 `sqlite3.connect()` 分散在大量模块，数据访问边界不清。
3. 多个采集入口并存，缺少统一任务状态、幂等和日志。
4. 三个核心数据库角色不清，代码直接跨库访问。
5. 时区、去重、球队映射逻辑重复分散。

### P2：中长期会限制产品化的点

1. SQLite 承担的数据规模和并发职责越来越重。
2. JSON 存储和赔率宽表缺少 serving 层转换。
3. 临时修复脚本多，缺少正式 migration 和质量闸门。
4. 前端缺少统一路由、状态管理和数据新鲜度模型。
5. 数据源限流、失败重试、健康监控还没有形成统一基础设施。

## 十、建议整改顺序

### 第一阶段：先止血

1. 修复前端搜索路由和 API 重复定义。
2. 取消页面挂载自动同步，改成明确按钮或后端定时任务。
3. 建立比赛状态标准化映射，把 `Finished`、`finished`、`FINISHED`、`FT` 等统一到 normalized status。
4. 明确竞彩链路最小闭环：比赛、赔率、预测、赛果、验证，先保证 SPF 一条链路完整。
5. 为关键接口补最小 contract 测试，确保前端调用的路由和参数真实存在。

### 第二阶段：收敛后端边界

1. 把 `main.py` 中的业务接口迁移到对应 router。
2. 拆分 `analytics/routes.py`，按领域拆成 elo、xg、odds、weather、tracking、news 等 router。
3. 把 `lottery.py` 中的同步、回填、分析、查询拆到 service/repository。
4. 统一数据库连接和配置读取，禁止新增硬编码数据库路径。
5. 建立统一 task 表，记录采集、同步、回填、验证、分析任务。

### 第三阶段：治理数据层

1. 明确 `football_v2.db`、`unified_football.db`、`oddsfe_merged.db` 的角色。
2. 设计 raw/staging/normalized/serving/model_dataset 分层。
3. 把实体映射、状态枚举、时间规范做成入库前校验。
4. 对大宽表和 JSON 表建立特征抽取或视图层，不直接让业务代码到处查。
5. 评估 PostgreSQL、DuckDB、Parquet 等更适合中长期分析和训练的存储方案。

## 十一、总结

这个项目最有价值的地方是：数据源多、足球分析方向明确、已经积累了大量真实数据，并且有从采集到预测再到复盘的闭环意识。

当前最大的问题不是“功能不够”，而是功能和数据增长太快，架构主线没有及时收敛。前端有路由和状态割裂，后端有大文件和直接连库，采集有多入口和脚本化倾向，存储有多库并存和质量不统一。

我的建议是：不要马上大重构，先修 P0 的明确 bug 和数据口径问题；然后把主链路固定下来；最后再治理存储分层和采集调度。这样可以保住已有数据资产，同时逐步把项目从“强大的个人工作台”推向“稳定的足球数据产品”。

## 十二、当前落地进展

这份分析之后，项目已经开始往“比赛驱动情报工厂”收敛，核心变化是：

1. 以单场比赛为入口生成 `intelligence_job`，再分发到各个采集模块。
2. 伤停、首发、新闻、天气、赔率、Elo、FIFA 排名、最近状态、世界杯上下文都进入同一套 package。
3. 对缺失信息不再静默失败，而是写入 `fallback_used`、`gaps` 和 `confidence`，方便后续复盘训练。
4. `tournament_context` 不再只靠 standings，而是结合 league rules、同系列赛程、当前赛事历史和国家队近况。
5. 后台定时任务已经切到北京时区，并把 injuries/lineup 也一起纳入刷新。
6. 新增 `intelligence_reviews` 复盘闭环，可从竞彩赛果和验证表自动生成预测、实际结果、命中情况、归因和证据快照。
7. 每天凌晨验证后会自动把上一北京日期的比赛同步成情报复盘，后续可直接用于训练样本和错误归因统计。
8. 新增已结束赛事回填能力：可把有赛果/验证的历史竞彩批量生成 intelligence job、package 和 review。
9. 新增训练样本读取能力：可直接读取 settled review、情报包摘要、requirement 状态、artifact 置信度和归因。

当前已实跑回填 `2026-05-24` 至 `2026-06-11` 的历史竞彩，生成 45 条已结算训练样本：其中 25 条正确，20 条错误；错误中包括 9 条 `low_strict_data_coverage`，这类样本很适合后续专门分析“弱证据/低严格覆盖率时模型为什么容易翻车”。

训练样本概览接口已经可以直接聚合这些样本：当前 45 条 settled 样本整体准确率约 55.6%；严格覆盖率 `50-69` 桶准确率约 30.0%，`70-84` 桶准确率约 64.7%。这说明后续不是盲目加模型，而是要优先补齐会影响严格覆盖率的字段，例如伤停、预计首发、市场变化、天气/场地等证据质量。

同时已经支持把训练样本导出为 JSONL，默认落在 `data/intelligence_exports/`，适合后续直接给训练脚本、Notebook 或模型评估流程使用。

今天补上的前端 `情报中枢` 更适合定位为后台诊断和训练面板，而不是替代体彩中心的主入口。真正的每日主驾驶舱仍然应该是 `体彩中心`：比赛卡片、预测、赛果和单场详情都应该继续在那里完成；情报层要嵌入体彩中心，为每场比赛提供证据覆盖率、缺失项、来源质量和赛后错因归因。

后端 `GET /api/v1/intelligence/jobs` 也补了前端需要的轻量字段：`package_completeness`、`strict_completeness`、`average_confidence`、`missing_required`。这让列表页不用逐场打开详情，也能直接看到哪场比赛已经可以分析，哪场只是用了兜底证据，哪场还需要补资料。

后续重点仍然不变：先把世界杯/国家队分析稳定住，再逐步扩展到联赛分析。

## 十三、长期目标再确认

这个项目最终不是一个单次预测工具，而是一个持续沉淀的足球数据与分析资产库。

它要把多渠道数据不断纳入同一条数据链路里：实时采集、历史回填、证据快照、赛后复盘、训练样本、模型校准、Agent 接入、统一查询。

核心原则是：

1. 实时数据一旦过了比赛，就转成历史证据和训练样本。
2. 体彩中心是主驾驶舱，负责展示比赛卡片、预测、赛果和详情。
3. 情报层负责证据覆盖、缺失项、来源质量、错因归因。
4. 云端数据库是生产资产，只能备份后迁移和增量更新，不能被本地库直接覆盖。
5. 历史样本越多，分析越稳，训练越强，解释越清楚。

## 十四、本次方向校正

当前主入口已经明确回到 `体彩中心`：每天比赛卡片、预测、赛果、单场详情仍然是核心工作流。`情报中枢` 不再作为主导航入口，而是保留为后台诊断/训练面板。

体彩中心应该直接承载情报层的关键结果：

1. 卡片上看到这场比赛的证据覆盖率、严格覆盖率和缺失项。
2. 详情里看到情报包、来源、置信度、兜底证据和下一步补采建议。
3. 分析结论和数据质量放在一起看，避免低证据覆盖的预测看起来和高证据覆盖的预测一样可靠。
4. 云端同步遵守 `云端同步与数据资产整改方案.md`：同步代码，不覆盖生产数据库。
